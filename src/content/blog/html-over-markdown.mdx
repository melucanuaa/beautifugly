## 为什么 HTML 更好

### 1. 信息密度高太多了

Markdown 能表达的东西很有限——标题、列表、表格、代码块，基本就这些。而 HTML 可以做：

- **对比表格** — 多个方案并排展示，优劣一目了然
- **SVG 图表** — 流程图、架构图，直接内嵌
- **代码高亮** — 带行号、带标注
- **交互控件** — 滑块调参数、下拉切换视图

Thariq 原话说得很有意思：没有 HTML 的时候，Claude 会用 Unicode 字符"估算颜色"来画图。有了 HTML，这些都不需要了。

### 2. 你真的会读 HTML，但不会读 Markdown

这一点扎心了。写过 AI 生成的长文档的人都有体会：

<div style="background:var(--color-bg-card);border-radius:10px;padding:14px;margin:1rem 0;font-size:0.82rem;border-left:3px solid var(--color-accent)">
  <strong>原话</strong>："我发现自己超过 100 行的 Markdown 根本读不下去，更别说让团队里其他人读了。"
</div>

HTML 有标签页（Tabs）、折叠块（可展开/收起）、视觉层级、链接跳转。一份 500 行的规格文档，用 HTML 渲染出来可能只要滚动两三屏就看完了。

### 3. 分享方便

Markdown 文件发给别人，对方还得装个渲染器。HTML？丢到任何地方，浏览器打开就能看。Thariq 的做法是上传到 S3 然后发链接。

> "一份规格文档、报告、或者 PR 说明，如果是 HTML 格式，被别人真正打开阅读的概率要高得多。"

### 4. 交互性：AI 输出 → 人调整 → 喂回 AI

这是最酷的部分。HTML 支持滑块、输入框、下拉菜单这些交互元素。典型的工作流：

## 怎么开始

不需要任何特殊配置，直接跟 AI 说就行：

```text
帮我做一个 HTML 文件，对比这三种数据库方案的优劣
```

```text
把这份 PR 的 diff 渲染成一个带标注的 HTML 页面
```

```text
做一个交互式的 HTML，让我能用滑块调整动画参数
```

就这么简单。生成的 `.html` 文件双击就能在浏览器里打开。

## 唯一的缺点

  <strong>Huǎn存</strong>：HTML 的 Git diff 很难读，比 Markdown 差很多。这是目前最大的短板。如果你需要对生成内容做版本控制，Markdown 还是有优势的。

## 我的看法

Thariq 最后说了一句很打动我的话：

> "用 HTML 之后，我比以前任何时候都更'在回路里'——与其让 Claude 自己做决定，HTML 让人始终保持参与。"

这和 Vibe Coding 的理念很像。AI 是工具，不是替代品。你需要能看懂它在做什么、能调它做出来的东西、能把它输出的结果继续喂回去。HTML 恰好提供了这种能力。

他还提到，用 Claude Code 生成 HTML "就是更有趣"。这点我同意——看到一个漂亮的、可交互的页面，比看一大段文字舒服多了。

**试试看吧，下次让 AI 输出 HTML。**
